# PRD-001: Despliegue AWS — BlendX Study Planner

## Problem Statement

La aplicacion BlendX Study Planner actualmente solo funciona en entorno local via `docker compose`. No existe infraestructura en produccion, pipeline de despliegue, ni separacion entre configuracion de desarrollo y produccion. Cada miembro del equipo tiene que levantar manualmente el proyecto para probarlo, y no hay un entorno compartido donde validar cambios antes de integrarlos.

## Solution

Desplegar la aplicacion en AWS usando infraestructura como codigo (CDK TypeScript) con los siguientes componentes:

- **Frontend (React/Vite):** AWS Amplify conectado al repositorio de GitHub, con build automatico al hacer merge a `main` y servido via CDN.
- **Backend (FastAPI + Docker):** ECS/Fargate con ALB publico, imagen Docker desde ECR, desplegado automaticamente por GitHub Actions.
- **Base de Datos (PostgreSQL 16.4):** RDS en subredes privadas, single-AZ, accesible solo desde Fargate.
- **Secretos:** Secrets Manager con un unico secreto JSON que agrupa `DATABASE_URL`, `JWT_SECRET`, y `OPENAI_API_KEY`.
- **CI/CD:** GitHub Actions con autenticacion OIDC a AWS que construye, pushea a ECR, y actualiza ECS al mergear PRs a `main`.

## User Stories

### Infraestructura como Codigo

1. As a DevOps engineer, I want to define all AWS resources in CDK TypeScript, so that I can version, review, and reproduce infrastructure consistently.
2. As a developer, I want the CDK stacks separated into NetworkingStack and AppStack, so that I can modify network topology without risking application resources.
3. As a DevOps engineer, I want to run `cdk deploy` once to create the entire environment, so that setup is a single command.
4. As a security reviewer, I want Security Groups defined as code, so that I can audit network access rules before deployment.

### Base de Datos

5. As a backend developer, I want PostgreSQL 16.4 with `pgvector` extension support, so that the database is compatible with the local development version and ready for future AI features.
6. As a security engineer, I want RDS deployed in private subnets with access restricted to the Fargate service only, so that the database is never exposed to the internet.
7. As a cost-conscious team member, I want single-AZ deployment with minimal instance size, so that database costs stay under budget during early stages.

### Backend en ECS/Fargate

8. As a backend developer, I want the Docker image built and pushed to ECR automatically on merge to main, so that I never manually upload images.
9. As a backend developer, I want Fargate to restart with the new image after each push, so that production always runs the latest merged code.
10. As a DevOps engineer, I want the Fargate task to read its environment variables from Secrets Manager at startup, so that no credentials are hardcoded in task definitions.
11. As a monitoring engineer, I want CloudWatch Logs enabled with 7-day retention, so that I can debug issues without retaining logs indefinitely.
12. As a developer, I want the `/health` endpoint used as the ALB health check, so that unhealthy tasks are automatically replaced.

### Load Balancer

13. As a user, I want HTTPS enforced via HTTP-to-HTTPS redirect on the ALB, so that traffic to the API is always encrypted.
14. As a DevOps engineer, I want the ALB to use an ACM certificate with DNS-validated auto-renewal, so that certificates never expire unexpectedly.
15. As a developer, I want all traffic routed to the Fargate target group, so that routing configuration is simple and predictable.

### Secretos

16. As a security engineer, I want database credentials auto-generated by CDK and stored in Secrets Manager, so that no passwords exist in code or config files.
17. As a developer, I want `OPENAI_API_KEY` stored in Secrets Manager as a placeholder to be filled manually, so that external API keys are never committed to the repository.
18. As a backend developer, I want all environment variables grouped in a single JSON secret, so that the task definition references one ARN instead of multiple.

### Frontend en Amplify

19. As a frontend developer, I want Amplify to auto-build on merge to `main`, so that frontend deployments are hands-off after code review.
20. As a frontend developer, I want `VITE_API_URL` configurable as an Amplify environment variable, so that the frontend knows the backend ALB DNS without rebuilding.
21. As a developer, I want the `amplify.yml` configuration committed to the repository, so that Amplify settings are version-controlled.

### CI/CD Pipeline

22. As a developer, I want CI/CD triggered only when a PR to `main` is merged and only when backend files changed, so that irrelevant changes dont trigger unnecessary builds.
23. As a security engineer, I want GitHub Actions to authenticate to AWS via OIDC (no long-lived access keys), so that credentials are never stored in repository secrets.
24. As a developer, I want the pipeline to build the Docker image, tag it with the commit SHA, push to ECR, and force a new ECS deployment, so that each deploy is traceable to a specific commit.

### Documentacion

25. As a new team member, I want a README section explaining the bootstrap process step by step, so that I can set up the infrastructure myself without tribal knowledge.
26. As a future maintainer, I want ADRs documenting key trade-off decisions, so that I understand why certain choices were made.

## Implementation Decisions

### Modulo 1: CDK IaC (`infra/`)

- **Lenguaje:** TypeScript CDK.
- **Stacks:** `NetworkingStack` (VPC, subnets, Internet Gateway) y `AppStack` (RDS, ECR, ECS Cluster, Task Definition, Fargate Service, ALB, Secrets Manager, Security Groups, IAM Roles).
- **Region:** `us-east-1` (default).
- **VPC:** 2 Availability Zones, 2 public subnets (para ALB y Fargate) y 2 private subnets (para RDS). CIDR `/16`.
- **RDS:** PostgreSQL 16.4, `db.t4g.micro`, 20 GB gp3 encrypted, single-AZ, en subredes privadas, credenciales via `rds.DatabaseSecret`.
- **ECR:** Un repositorio con `lifecycleRule` que retiene las ultimas 10 imagenes.
- **ECS:** Cluster Fargate, Task Definition con 256 CPU / 512 MB, platform `LINUX` (x86_64), 1 tarea sin auto-escalado, CloudWatch Logs con 7 dias de retencion.
- **ALB:** `internetFacing`, listener HTTP (:80) redirige a HTTPS (:443), certificado ACM auto-generado, DNS auto-asignado (sin dominio propio).
- **Secrets Manager:** Un secreto JSON con tres campos: `DATABASE_URL`, `JWT_SECRET`, `OPENAI_API_KEY`. `DATABASE_URL` se construye con credenciales del RDS. `JWT_SECRET` se genera automaticamente. `OPENAI_API_KEY` es un placeholder.
- **Security Groups:**
  - `ALB-SG`: Ingress :80 y :443 desde `0.0.0.0/0`.
  - `Fargate-SG`: Ingress :8000 solo desde `ALB-SG`. Egress a internet para ECR y OpenAI API.
  - `RDS-SG`: Ingress :5432 solo desde `Fargate-SG`.
- **IAM Task Role:** Permiso `secretsmanager:GetSecretValue` sobre el ARN del secreto.
- **Concesion registrada:** Fargate se despliega en subredes publicas con IP publica (en lugar de usar NAT Gateway en subredes privadas) para reducir costos mientras se mantiene capacidad de llamar a la API de OpenAI. Se compensa con un Security Group restrictivo que solo permite trafico entrante desde el ALB.

### Modulo 2: Pipeline CI/CD (`.github/workflows/deploy.yml`)

- **Trigger:** `pull_request` evento `closed` en rama `main`, con filtro `merged == true` y `paths: ['backend/**']`.
- **Jobs:**
  1. `build-and-push`: Checkout del repo, configuracion de credenciales AWS via OIDC (`aws-actions/configure-aws-credentials`), login a ECR, build Docker con tag `${{ github.sha }}`, push a ECR.
  2. `deploy`: Obtener la task definition actual de ECS, registrar una nueva revision con la nueva imagen, forzar `update-service --force-new-deployment`.
- **AWS Auth:** OIDC (`GitHub OIDC provider` + IAM role con `AssumeRoleWithWebIdentity`). No se almacenan access keys en Secrets del repositorio.
- **Permisos IAM para el role de GitHub Actions:** `ecr:GetAuthorizationToken`, `ecr:BatchCheckLayerAvailability`, `ecr:GetDownloadUrlForLayer`, `ecr:PutImage`, `ecr:InitiateLayerUpload`, `ecr:UploadLayerPart`, `ecr:CompleteLayerUpload`, `ecs:DescribeTaskDefinition`, `ecs:RegisterTaskDefinition`, `ecs:UpdateService`, `ecs:DescribeServices`, `iam:PassRole` (limitado al task execution role ARN).

### Modulo 3: Amplify Frontend (`amplify.yml`)

- **Archivo:** `amplify.yml` en la raiz del repositorio.
- **Framework:** Vite (Amplify lo detecta del `package.json`).
- **Build output:** `frontend/dist/`.
- **Fases:** `preBuild` ejecuta `cd frontend && npm ci`, `build` ejecuta `npm run build` (que a su vez corre `tsc && vite build`).
- **Cache:** `frontend/node_modules/`.
- **Variable de entorno:** `VITE_API_URL` configurada en la consola de Amplify, apuntando al DNS del ALB (`http://<alb-dns>`). En desarrollo local, Vite usa proxy internamente, por lo que esta variable solo se necesita en produccion.
- **Rama de despliegue:** `main`.

### Modulo 4: Dockerfile de Produccion (`backend/Dockerfile`)

- **Cambio unico:** Eliminar `--reload` del CMD de uvicorn.
- **CMD resultante:** `uv run uvicorn app.main:app --host 0.0.0.0 --port 8000`.
- **Razon:** En desarrollo, `docker-compose.yml` ya sobreescribe el CMD con `--reload` via `command`. El Dockerfile base queda como configuracion de produccion sin duplicar archivos.
- **Sin multi-stage:** La imagen de produccion es identica a la de desarrollo (cambio en CMD), por lo que multi-stage build no aporta beneficio en este momento.

### Modulo 5: Documentacion

- **README.md:** Nueva seccion "AWS Deployment" que documenta:
  - Prerrequisitos (AWS CLI configurado, CDK bootstrap, cuenta AWS).
  - Paso 1: `cdk deploy NetworkingStack AppStack`.
  - Paso 2: Llenar `OPENAI_API_KEY` en Secrets Manager (consola AWS).
  - Paso 3: Configurar `VITE_API_URL` en la consola de Amplify.
  - Costo estimado mensual (~$50-70/mes).
  - Diagrama de arquitectura (ASCII o referencia a la seccion de arquitectura).
- **CONTEXT-MAP.md:** Archivo raiz que mapea los contextos `backend/` y `frontend/`.
- **backend/CONTEXT.md:** Glosario de dominio del backend (User, StudyPlan, StudyTask, JWT, etc.).
- **frontend/CONTEXT.md:** Glosario de dominio del frontend (Page, Modal, Auth flow, etc.).
- **docs/adr/0001-fargate-public-subnet.md:** Justifica la concesion de Fargate en subred publica con IP publica en lugar de NAT Gateway.
- **docs/adr/0002-single-dockerfile.md:** Justifica mantener un solo Dockerfile sin multi-stage.

## Testing Decisions

- **CDK:** Sin tests de snapshot para este MVP. Los stacks se validan manualmente ejecutando `cdk synth` y verificando la salida de CloudFormation.
- **Pipeline:** Sin tests automatizados sobre el workflow. Se valida con el primer PR mergeado a `main`.
- **Backend:** Los tests pytest existentes permanecen sin cambios. No se requieren tests adicionales para la infraestructura.
- **Que hace un buen test:** Solo se testea comportamiento externo (endpoints HTTP), no implementacion interna de infraestructura.

## Out of Scope

- Auto-escalado de Fargate (se hara en iteracion futura).
- Multi-AZ para RDS (se activara cuando la aplicacion tenga trafico real).
- Dominio propio con Route53 (se usara DNS auto-asignado del ALB).
- Pipeline de CI/CD para el CDK mismo (el CDK se ejecuta manualmente).
- Entorno de staging separado (solo produccion por ahora).
- Soporte para ARM/Graviton (se evaluara cuando las dependencias sean compatibles).
- Configuracion de WAF o proteccion DDoS.
- Monitoreo con CloudWatch Alarms o dashboards personalizados.
- Backup automatizado de RDS mas alla del backup automatico por defecto.

## Further Notes

- El bootstrapping inicial (primer `cdk deploy`) debe hacerse desde una maquina local con AWS CLI configurado y permisos de administrador.
- El OIDC provider de GitHub debe configurarse manualmente la primera vez en IAM (o via CDK en un stack de bootstrap).
- Los costos estimados asumen uso minimo continuo (~$50-70/mes total: RDS ~$15, Fargate ~$15, ALB ~$20, Secrets Manager ~$0.50, CloudWatch ~$2-5, Amplify free tier).
- El `OPENAI_API_KEY` en Secrets Manager debe llenarse manualmente desde la consola de AWS. Si el backend intenta leerlo y no existe, debe fallar con un error claro al iniciar (no en runtime).
